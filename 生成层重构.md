**你现在需要的是一轮“契约下沉 + 玩法运行时库 + 语义验证”的中等重构。现有链路的 APRD → mechanics.yaml → specs → implementation-contract → verify_all 方向是对的，接管文档里也明确把它定义为“不要从 PRD 散文直接写代码，而是把玩法语义、素材语义、启动契约、验证证据前置成可检查中间工件”。问题在于：**这些中间工件目前还没有完全变成代码生成时不可绕过的执行真值**。

## 1. 这条链路真正的问题在哪里

### 问题 A：`mechanics.yaml` 验证的是“规格可行”，不是“生成代码真的实现了这个机制”

你已经有 `check_mechanics.js`，它会读取 `specs/mechanics.yaml`，加载 primitive reducer，跑 simulation scenarios，检查 invariants、hard-rule 覆盖、win 可达等；这一步能证明 mechanics DAG 自身结构成立。但是 Phase 4 里 codegen 仍然是让 LLM 按 spec 自己写实现，只要代码里有 `@primitive(...)` 注释、`window.__trace.push({rule: ...})` 之类证据，就能通过相当一部分静态门禁。`check_implementation_contract.js` 已经会检查 mechanics node 是否有 `@primitive(...)` 注释、rule trace push 点、required asset 消费等，但这仍然主要是“代码里有落点”，不是“代码行为和 reducer 等价”。

这就是“玩法不是先天生成好，而是后验修补”的根因。**Phase 3 证明了玩法蓝图，Phase 4 仍然在让模型重新发明玩法实现。**

最典型的例子就是 Pixel Flow：需求明确要求小猪必须按当前位置处理当前方向第一个块，不能全屏索敌、不能穿透异色块、不能一次攻击多个同色块。`ray-cast@v1` 的语义也明确禁止全局 `blocks.find(b => b.color === pig.color)`，要求命中结果依赖 source.position，并遵守 first-hit / no-penetration。但如果 codegen 写了一个“看起来类似”的全局查找，当前链路不一定能在 Phase 4 直接打掉。

### 问题 B：产品验证还偏“覆盖率”，不是“语义断言”

`check_playthrough.js` 现在已经比过去强很多：profile 不能写 `expect`，必须有真实 click/press/fill，不能只靠 `window.gameTest` 代替真实输入，并会检查 `window.__trace` 覆盖 PRD rule。但它的核心真相仍主要是“trace 覆盖率 + 无 runtime error + 有真实点击”。这能证明“某条规则被执行过”，不能证明“攻击目标就是当前位置对应方向的第一个目标块”。

你自己的诊断文档也指出过这个问题：trace 只能证明代码执行过某个分支，不证明 `before/after` 符合 PRD；比如不能证明小猪靠近同色方块时一定攻击、异色第一块会阻挡后方同色块、每次只攻击当前方向第一个目标等。

所以 Phase 5 现在比以前更难被“纯手写绿色报告”骗过，因为 `verify_all.js` 已经存在并会从真实退出码写 `eval/report.json`；但它还会被“补 trace / 触发分支 / profile 驱动捷径”绕过。你缺的是**语义 oracle**，不是更多普通 UI 测试。

### 问题 C：素材链路有 contract，但缺少一等公民的“视觉语义槽”

你已经把资产链路设计成 `asset-strategy -> assets.yaml -> implementation-contract -> registry manifest -> business code 消费 -> check_asset_usage / boot runtime 证据`，并且明确“manifest 注册不等于业务消费”。`generate_implementation_contract.js` 也会根据 assets 生成 `asset-bindings`，包含 role、asset-kind、binding-to、must-render、allow-fallback 等字段。

但现在的问题是：**role / asset-kind 多数还是从 id、usage、source 路径推断出来的，而不是从一个强 schema 里的 `semantic-slot` 决定。**这会导致“绑定到 block”不等于“它长得像 block”，“被业务消费”不等于“被用在正确的视觉位置”。你接管文档里已经补了色块规则：如果实体是色块/方块/目标块，且玩法依赖 color 字段，必须用 generated / graphics-generated / inline-svg，并写 `visual-primitive: color-block`，不能用 coin、gem、dungeon tile、button 绑定给抽象色块。但这个规则还应继续下沉成 schema 和 checker，而不是主要靠 expander prompt。

### 问题 D：主流程仍然太依赖 agent “自觉执行”

当前 `SKILL.md` 已经把流程写得很完整：Phase 2.5 必须产 `spec-clarifications.md`，Phase 3.0 先产 `mechanics.yaml`，Phase 3 gates 跑 `check_asset_selection / check_implementation_contract / check_mechanics`，Phase 4 gates 跑 `check_mechanics / check_project / check_game_boots`，Phase 5 跑 `verify_all`。但这些大多还是 Markdown SOP + 主 agent 行为约束。当前框架报告也点名说：很多约束写在 `SKILL.md`、agent prompt 和 Markdown SOP 中，脚本只覆盖其中一部分；如果主 agent 漏跑脚本、漏解析子 agent JSON，或失败时继续往下走，链路仍会漂移。

这不是“LLM 不够聪明”的问题，而是**编排层还没有完全程序化**。

### 问题 E：primitive 库覆盖了核心运动/空间/逻辑，但还缺少“生命周期/槽位/容量”类原语

当前 Stage 1 primitive 有 parametric-track、grid-step、grid-board、ray-cast、neighbor-query、predicate-match、resource-consume、fsm-transition、win-lose-check、score-accum 等。这能覆盖 Pixel Flow 的轨道、射线、匹配、扣耐久、胜负。但 Pixel Flow 的等待槽回收、传送带容量、小猪派出/回收/消失、剩余数值保留，是玩法核心，不是 UI 附属规则。测试 prompt 明确写了等待槽回收、容量上限、多只小猪同时在传送带、数值耗尽离场、剩余数值回到底部等待槽等要求。

这些现在多半会落到 event-graph / codegen 的自由实现里。只要没有 `slot-pool / capacity-gate / entity-lifecycle / recycle` 这类 primitive，模型就会在最容易出错的资源生命周期上自由发挥。

## 2. 补丁级修改还能不能解决？

**prompt 补丁不够。脚本/契约级补丁能明显改善。完整推倒重构没必要。**

我的建议是把改动分成三类：

| 类型                                       | 是否值得做 | 作用                                            |
| ------------------------------------------ | ---------- | ----------------------------------------------- |
| 继续补 agent prompt 文案                   | 低         | 会短期改善，但模型仍可能绕过或遗忘              |
| 给现有 checker 加规则                      | 中         | 能快速堵住 forceWin、素材假消费、色块误绑等问题 |
| 把 primitive 实现变成运行时库 + 语义 probe | 高         | 这是把链路和通用 app 生成拉开差距的关键         |
| 改成每个 genre 一套厚模板                  | 不建议     | 短期稳定，长期覆盖不了长尾玩法，维护成本高      |

所以不是“补丁级 vs 全重构”二选一，而是：**保留当前流水线，重构最关键的三个断层：编排层、primitive runtime、语义验证层。**

## 3. 最重要的改法：不要让 codegen 重新实现 primitive

你现在最该做的一件事是：**把 primitive reducer 扩展成 browser 可用的 runtime primitive 库。**

现在的机制是：

```
mechanics.yaml
  -> check_mechanics 用 reducer 证明 DAG 可行
  -> codegen 读 spec.md，自己写 JS 实现
```

建议改成：

```
mechanics.yaml
  -> check_mechanics 用 reducer 证明 DAG 可行
  -> codegen 直接 import / copy 官方 primitive runtime
  -> LLM 只负责 wiring、渲染、UI、layout、asset binding
```

新增目录可以是：

```
game_skill/skills/references/mechanics/runtime/
├── parametric-track.runtime.mjs
├── ray-cast.runtime.mjs
├── grid-board.runtime.mjs
├── predicate-match.runtime.mjs
├── resource-consume.runtime.mjs
├── slot-pool.runtime.mjs
├── capacity-gate.runtime.mjs
└── index.mjs
```

Codegen 阶段不再让模型手写 `raycastFromPig()`，而是生成类似：

```
import {
  updateParametricTrack,
  projectRectLoopToGrid,
  rayCastGridFirstHit,
  predicateMatch,
  consumeResource,
} from './mechanics-runtime/index.mjs';

function onTrackEnterSegment(pig, segmentId) {
  const source = projectRectLoopToGrid(pig, trackParams);
  const hit = rayCastGridFirstHit({
    source,
    direction: directionFromSegment(segmentId),
    targets: state.blocks,
    stopOn: 'first-hit',
  });

  if (!hit) return traceRule('ray-miss', { pig });

  if (!predicateMatch(pig, hit.target, ['color'])) {
    return traceRule('color-mismatch-blocked', {
      pig,
      firstTarget: hit.target.id,
    });
  }

  consumeResource({
    source: pig,
    target: hit.target,
    sourceField: 'ammo',
    targetField: 'durability',
    amount: 1,
  });
}
```

这会直接解决“模型写成全局自动索敌”的大问题。`ray-cast@v1` spec 已经明确 grid 模式要从 source 按 dx/dy 步进格子查询，禁止全局同色搜索；`parametric-track@v1` 也已经明确棋盘外圈要用 rect-loop，并且 canvas 绘制禁止用 `ctx.arc()` 画成圆。现在要做的是：**让这些规则由 runtime 函数执行，而不是让模型重新理解一遍。**

这不是厚模板。它是“正交 primitive runtime 库”。它比 genre template 更适合长尾小游戏。

## 4. 第二个关键改法：新增 `check_runtime_semantics.js`

`check_mechanics.js` 证明 specs 成立；`check_playthrough.js` 证明 UI 能驱动；中间缺一层：

```
check_runtime_semantics.js
```

它应该做三件事：

第一，读取 `mechanics.yaml`、`rule.yaml`、`event-graph.yaml`，找出高风险 primitive 组合，比如：

```
parametric-track + ray-cast(grid) + predicate-match + resource-consume
slot-pool + capacity-gate + entity-lifecycle
resource-consume + win-lose-check
```

第二，在浏览器里注入固定 probe scenario，但仍通过真实输入触发关键动作。例如 Pixel Flow 至少要自动生成这些 probe：

```
probe.raycast.first-hit.same-color:
  pig 在 top col=2
  blocks: row0 col2 red, row1 col2 red
  触发一次经过
  期望：只攻击 row0 col2，不攻击 row1 col2

probe.raycast.no-penetration.mismatch:
  pig red 在 top col=2
  blocks: row0 col2 blue, row1 col2 red
  触发一次经过
  期望：两个块都不被攻击

probe.raycast.position-dependent:
  pig 在 top col=1 与 top col=4 各跑一次
  期望：命中列不同，禁止固定从 col=0 扫描

probe.recycle.remaining-ammo:
  pig ammo=3，攻击一次后完成 loop
  期望：回等待槽，ammo=2，仍可再次派出

probe.capacity-gate:
  conveyor capacity=2
  连续点击 3 只 pig
  期望：第 3 只留在等待槽，不进入轨道
```

第三，不只看 trace 是否出现，而是检查 trace 的 before/after：

```
window.__trace.push({
  rule: 'attack-consume',
  primitive: 'resource-consume@v1',
  node: 'attack-consume',
  event: { type: 'match.hit', sourceId: 'pig-1', targetId: 'block-0-2' },
  before: {
    pig: { id: 'pig-1', ammo: 3, gridPosition: { row: -1, col: 2 } },
    target: { id: 'block-0-2', durability: 1, color: 'red' }
  },
  after: {
    pig: { id: 'pig-1', ammo: 2 },
    target: { id: 'block-0-2', alive: false, durability: 0 }
  }
});
```

然后 checker 用 primitive reducer 或 runtime primitive 复算 delta。这样 `window.__trace.push({rule:'attack'})` 这种“补 trace 过关”会失效。

## 5. 第三个关键改法：素材必须从 `binding-to` 升级到 `semantic-slot`

当前素材链路已经能检查 local-file 是否存在、manifest 是否覆盖、required asset 是否消费、button 是否误绑文字承载 UI 等。但你真正想解决的是“合适素材出现在合适位置”。这需要把 `semantic-slot` 作为一等字段。

建议 `assets.yaml` 里每个核心视觉条目必须长这样：

```
images:
  - id: block-red
    binding-to: block
    semantic-slot: color-block
    visual-primitive: color-block
    type: graphics-generated
    color-source: entity.color
    usage: 中央棋盘目标块，颜色是玩法字段，不允许具象 tile/coin 替代

  - id: pig-unit
    binding-to: pig
    semantic-slot: colored-unit
    type: local-file
    source: assets/library_2d/...
    color-overlay: entity.color
```

Catalog 也要从 pack-level 语义升级到 asset-level 或至少 family-level 语义：

```
assets:
  - path: tiles/dungeon/tile_0078.png
    semantic-tags: [dungeon-tile, terrain]
    allowed-slots: [terrain-cell, background-tile]
    disallowed-slots: [color-block, target-block, button]

  - path: ui-pixel/button_001.png
    semantic-tags: [button, ui-control]
    allowed-slots: [button]
```

然后新增或强化 checker：

```
check_asset_selection.js:
  - core entity 不能 role=decorative
  - binding-to ∈ visual-core-entities 时必须 semantic-slot 合法
  - color 字段参与 rule 的 entity，semantic-slot 必须是 color-block / colored-unit / colorable-token
  - semantic-slot=color-block 时，只允许 generated/inline-svg 或 catalog allowed-slots 包含 color-block

check_implementation_contract.js:
  - core binding 的 must-render 必须 true
  - core binding 的 allow-fallback 必须 false
  - role=decorative 不得满足 visual-core-entities
  - asset-kind=tile/coin/icon 不得绑定 semantic-slot=color-block

check_asset_usage.js:
  - 不只检查 id 出现 + 某处有消费调用
  - 要求 registry.getTexture('asset-id') / drawImage(asset-id) / DOM img src 形成 per-asset 消费证据
  - runtime 记录 window.__assetUsage.push({id, bindingTo, semanticSlot, renderedAt})
```

你现在的接管文档已经写清楚“资产不是有引用就算用”，正确链路要从 PRD asset-strategy 到 assets.yaml、contract、registry、business code、runtime 证据一路贯通。下一步就是让 `semantic-slot` 成为这条链的主键。

## 6. 第四个关键改法：把主流程调度做成可执行 runner

现在有很多门禁，但“是否跑、何时跑、失败是否中止”仍然太依赖主 agent。建议新增：

```
game_skill/skills/scripts/run_phase_gate.js
game_skill/skills/scripts/run_case_pipeline.js
```

最小可行版不需要替代 LLM 生成内容，只负责**验收每阶段是否可进入下一阶段**：

```
node run_phase_gate.js cases/${PROJECT} --phase prd
node run_phase_gate.js cases/${PROJECT} --phase expand
node run_phase_gate.js cases/${PROJECT} --phase codegen
node run_phase_gate.js cases/${PROJECT} --phase verify
```

每个 gate 做固定动作：

```
phase=prd:
  - docs/game-prd.md exists
  - check_game_prd.js pass
  - profile skeleton exists
  - guardrails.md exists

phase=expand:
  - spec-clarifications.md exists
  - 7 specs exist
  - no specs/.pending residue
  - check_asset_selection pass
  - check_implementation_contract --stage expand pass
  - check_mechanics pass

phase=codegen:
  - game/index.html exists
  - generate_registry already reflected
  - check_mechanics pass
  - check_project pass
  - check_game_boots pass
  - check_runtime_semantics pass

phase=verify:
  - profile frozen
  - specs frozen
  - verify_all pass
  - report.generated_by === verify_all.js
```

这样即使 agent 文档漏读，脚本也会阻断。当前框架报告已经建议把 Phase 调度做成统一 runner，把“必须跑哪些脚本、哪些退出码阻断”从文档约束下沉到可执行编排。

## 7. 需要补的 primitive，不要靠 codegen 自由实现

针对你现在的典型 case，建议 Stage 2 先补这四个 primitive：

```
slot-pool@v1
  用于等待槽、背包槽、手牌槽。
  invariant: capacity、unique occupancy、preserve fields on return。

capacity-gate@v1
  用于传送带容量、场上单位上限、冷却派出。
  invariant: count(active) <= capacity。

entity-lifecycle@v1
  用于 spawn / active / return / despawn / dead。
  invariant: 状态转移闭合，不能同时在 waiting 和 active。

cooldown-dispatch@v1 或 input-dispatch@v1
  用于点击派出、输入到实体生命周期的绑定。
  invariant: 输入只能改变允许的目标实体，不能直接改 win/lose。
```

Pixel Flow 的等待槽和回收不是“表现层”，是核心策略循环。接管文档也把 mechanics.yaml 定义为玩法真值层：PRD 说想要什么，mechanics.yaml 说明玩法如何由 primitive DAG 表达，codegen 负责实现 DAG，check_mechanics 验证 DAG 成立。如果等待槽、容量、回收不在 DAG 里，它们就会落回“模型猜实现”。

## 8. 验证层还要补两个反作弊点

### 禁止 profile 里的捷径 mutator

`check_playthrough.js` 现在会检查真实 click，也会禁止 `expect`，但还应该扫描 profile 的 `eval` setup，禁止这些模式：

```
/force(?:Win|Lose|Pass|Fail)\s*\(/i
/window\.gameState\s*=/
/window\.gameState\.[\w.]+\s*=/
/Object\.assign\s*\(\s*window\.gameState/
/state\.(phase|status|score|win|lose|result)\s*=/
/__trace\.push\s*\(/
```

profile 可以驱动 UI，但不能写结果、不能补 trace、不能直接改核心状态。你现在文档里已经规定 profile 只负责驱动 UI，不能作为真相，禁止直接改 `window.gameState`，产品真相来自 `window.__trace`、runtime errors、asset errors 和脚本规则。这个规定需要在 `check_playthrough.js` 里变成硬正则或 AST 检查。

### 把 `window.gameTest` 分成 observer / driver / probe 三类

现在 `window.gameTest` 容易变成“测试捷径”。建议 event-graph 里声明：

```
test-hooks:
  observers:
    - getSnapshot
    - getTrace
  drivers:
    - clickStartButton
    - clickRetryButton
  probes:
    - resetWithScenario
    - stepTicks
```

规则：

```
playthrough profile:
  只允许真实 click/press/fill，observer 可读，禁止 probe。

runtime semantic probe:
  允许 resetWithScenario / stepTicks，但必须用固定 checker 生成的 scenario，不能由 profile 自由写。

delivery verify:
  只引用 verify_all report。
```

这样你既能做确定性语义测试，又不会让普通产品验收被 `forceWin()` 这类函数污染。

## 9. 可玩性问题不能只靠测试，要加 `solution-path` 和 solver

你说“最后面向测试过了，但一点都不可玩”，这里还有一个独立问题：**规则正确不等于关卡可玩**。

建议 `data.yaml` 必须包含：

```
solution-path:
  min-steps: 6
  max-steps: 18
  actions:
    - dispatch: pig-red-1
      expected-hit: block-0-2
    - recycle: pig-red-1
    - dispatch: pig-blue-1
      expected-hit: block-0-1

playability:
  solvable: true
  trivial-click-all: false
  meaningful-decisions-min: 3
  failure-path-exists: true
  no-softlock-after-valid-prefix: true
```

然后新增：

```
check_balance.js 或 check_level_solvability.js
```

对不同 primitive 组合做轻量验证：

```
board-grid puzzle:
  - 至少一条 solution-path 能 win
  - 不是任意点击都 win
  - 至少有一个错误顺序会导致资源变差
  - 所有合法前缀不应进入不可恢复软锁，除非 PRD 明确允许失败

reflex / action:
  - 目标出现频率、命中窗口、失败反馈、重试路径

edu-practice:
  - 题目数量、正确/错误反馈、渐进难度
```

这仍然不是厚 genre 模板，而是**基于 primitive 的可玩性 validator**。例如 Pixel Flow 可以从 solution-path 反向生成棋盘：先设计攻击顺序，再放置阻挡块和颜色，让关卡天然可解且有策略性。

## 10. 建议的修改顺序

### P0：先堵“过 checker 但不正确”的口子

1. 在 `check_playthrough.js` 增加 profile eval 禁止项：`forceWin/forceLose/direct gameState mutation/__trace.push`。
2. 新增 `check_runtime_semantics.js`，先只覆盖 Pixel Flow 高风险组合：`parametric-track + ray-cast(grid) + predicate-match + resource-consume`。
3. 在 `verify_all.js` 中把 `check_runtime_semantics.js` 放在 `check_playthrough` 前或后；失败则 report failed。
4. 在 `generate_implementation_contract.js / check_implementation_contract.js` 中强制 core entity 不能 role=decorative，core binding 必须 must-render。
5. 在 `check_asset_selection.js` 中强制 color-dependent block 使用 `visual-primitive: color-block` 或合法 `semantic-slot=color-block`。

### P1：把玩法实现从 LLM 手里收回来

1. 给前 5 个高频 primitive 做 browser runtime：`parametric-track / ray-cast / predicate-match / resource-consume / win-lose-check`。
2. 修改 `engine-codegen.md`：Codegen 必须 import runtime primitive，不允许手写 raycast/resource consume 算法。
3. 添加 `slot-pool / capacity-gate / entity-lifecycle` primitive，专门覆盖等待槽、容量、回收。
4. `check_implementation_contract.js` 不只查 `@primitive` 注释，还查对应 runtime primitive 是否被 import/called。

### P2：解决“正确但不好玩”

1. `data.yaml` 增加 `solution-path`。
2. 新增 `check_level_solvability.js`。
3. 建固定 benchmark：`test/word.md`、`test/pixel.md`、`test/roguelike.md`，每次链路变更后跑同一批样本。公开仓库里已有这三个 test prompt，其中 `pixel.md` 是非常适合做复杂机制回归的压力样本。

## 11. 最终建议

**不要推倒重来。**你现在的方向已经比“通用生成 app”多了 APRD、mechanics、contract、asset strategy、verify_all、freeze，这些都是正确的护城河。接管文档里也明确建议不要走“每种游戏一个厚模板”，而是走 `primitive mechanics + common game systems + engine templates + asset catalog + contract gates` 的组合路线。

但你要停止做两类低收益补丁：

```
1. 继续把 prompt 写得更长；
2. 继续在 Phase 5 发现 bug 后补 profile / 补 trace / 补静态规则。
```

真正该做的是：

```
mechanics.yaml 不只是 checker 输入，
而要变成 runtime implementation 的输入。

asset binding 不只是 source/path/id，
而要变成 semantic-slot + runtime rendered evidence。

playthrough 不只是 trace coverage，
而要变成 before/after semantic assertion。

主流程不只是 SKILL.md 约束，
而要变成 runner 阻断。
```

做完这几件事，你的链路才会和通用 app 生成拉开差距：通用 app 生成器能写 UI 和简单状态，但它不会有可组合的游戏 primitive runtime、机制级 oracle、素材语义槽、solver 级可玩性验证。

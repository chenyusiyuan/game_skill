针对素材乱用 / 不用，我的判断是：

**这部分不需要推翻整体链路，但需要把 asset 流程从“选文件 + 引用文件”升级成“视觉语义槽 + 渲染证据”。**
你现在的链路已经意识到“manifest 注册不等于业务消费”，接管文档里也明确说正确资产链路应是 `PRD asset-strategy -> assets.yaml -> implementation-contract -> registry manifest -> business code 真实消费 -> runtime 证据`，这条方向是对的。问题是现在 contract 还不够细，尤其缺少“这个素材应该承担什么视觉语义”的强约束。

## 1. 现状已经做对了什么

你现在不是完全没有素材门禁。已有几块是有效的：

`check_asset_selection.js` 已经检查 `assets.yaml`、`color-scheme.palette-id` 到 catalog asset-style 的映射、genre 合法性、本地文件是否存在、pack 的 suitable-styles / suitable-genres 是否匹配、selection-report 是否存在等。

它也已经开始处理核心问题：如果某个图片或 spritesheet 被识别成色块 / 目标块语义，会禁止使用具象 local-file，并要求 `generated / graphics-generated / inline-svg + visual-primitive: color-block`；同时 local-file 必须声明 `binding-to`，且 `binding-to` 要能对应 PRD 的 `@entity/@ui`。

`generate_implementation_contract.js` 会把 assets 转成 `asset-bindings`，并派生 `role`、`asset-kind`、`binding-to`、`render-as`、`must-render`、`allow-fallback`、`consumer` 等字段；`must-render` 目前主要由真实 binding、类型、是否 optional 状态决定。

`check_asset_usage.js` 也已经从旧的“整体消费率 60%”推进到“contract 里 must-render=true 的 required asset 必须 100% 被业务代码消费”，并且会排除 adapters、common、manifest 这类粘合文件，避免“只注册 manifest 不渲染”直接过关。

所以素材链路方向是对的。真正的问题是：**这些检查仍然偏静态、偏文件、偏粗粒度 binding。**

## 2. 现在为什么还会乱用素材

### 2.1 `binding-to` 太粗，不等于“用在正确位置”

现在素材主要绑定到 `@entity/@ui`，例如：

```
binding-to: block
binding-to: pig
binding-to: button-start
```

但这只能说明“这个素材和某个实体有关”，不能说明它承担的是：

```
block 的颜色填充？
block 的边框？
block 的破碎特效？
pig 的主体？
pig 的头像？
等待槽背景？
按钮背景？
HUD icon？
```

所以模型可以写出看似合法但语义错误的东西：

```
id: block-texture
binding-to: block
source: assets/library_2d/dungeon/tile_012.png
```

它绑定到了 `block`，文件也存在，pack 风格也可能匹配，但视觉语义仍然错。

### 2.2 `role` / `asset-kind` 目前主要靠字符串推断

`generate_implementation_contract.js` 现在会根据 id、usage、source、section 推断 `role` 和 `asset-kind`。例如包含 `button/按钮`推成 button，包含 `tile_` 或 `/tiles/` 推成 tile，包含 `character/player/enemy` 推成 sprite，不匹配则默认 decorative。

这会导致两个风险：

第一，模型可以通过改 id / usage 绕过语义。例如把 tile 写成 `block-surface`，role 就可能变得模糊。

第二，catalog 只告诉你 pack 层面的 style / genre 适配，不足以判断单个文件能否当作某个 gameplay entity。`tile_0013.png` 到底是地砖、墙、按钮、目标块，不能只靠路径和文件名判断。

### 2.3 `decor` 还是太宽松

`check_asset_selection.js` 现在会统计 local-file 里 `binding-to: decor` 的占比，超过 40% 只是 warning，不是 fail。这会留下一个偷懒通道：expander 可以把不好绑定的素材标成 decor，只要核心实体另有少量 asset，就可能继续往下走。

对小游戏来说，decor 不是不能存在，但**核心玩法实体不能靠 decor 模糊带过**。

### 2.4 pack coherence 也是 warning，不是硬约束

同一个 case 的 local-file 应该尽量来自同一个主 pack，当前脚本会检查最大 pack 覆盖率，但低于阈值时也是 warning，除非有 `selection-report.pack-mix-reason` 说明混包原因。这对“风格割裂”只能提醒，不能阻断。

如果目标是“可交付小游戏”，建议把这条按 `style-coherence.level` 分档：

```
strict      -> fail
flexible    -> warn
n/a         -> skip
```

教育卡牌、休闲 UI、棋盘 puzzle 通常应该至少 flexible；像 roguelike / pixel dungeon 可以允许同 pack 内混用 tile 和 sprite，但不应跨很多 pack 乱拼。

## 3. 现在为什么还会“不用素材”

### 3.1 目前 usage 检查仍然是“引用 + 全局消费函数”，不是 per-asset 消费

`check_asset_usage.js` 对每个 required asset 会看业务源码里是否出现 asset id 或 basename，然后再看业务源码里是否出现某类消费函数，比如 `drawImage`、`add.image`、`new Sprite`、`getTextureUrl` 等。

核心问题是：**`hasRef` 和 `hasConsumer` 没有绑定到同一次调用。**

也就是说，理论上下面这种代码可能骗过一部分静态检查：

```
const unused = 'pig-sprite';

const tex = registry.getTexture('background');
ctx.drawImage(tex, 0, 0);
```

因为 `pig-sprite` 这个 id 出现了，代码里也有 `getTexture/drawImage`，但它并没有证明 `pig-sprite` 被拿来渲染。当前代码确实是对每个 asset 做 `hasRef && hasConsumer`，但 `hasConsumer` 是在整份 businessSrc 上匹配，不是匹配 `getTexture('该 asset id')` 或 `drawImage(该 asset 的 texture)` 这一条链路。

### 3.2 “消费”不等于“可见”

就算代码里有：

```
this.add.image(x, y, 'pig-sprite');
```

仍然不一定说明素材在最终画面里可见。它可能：

```
alpha = 0
visible = false
scale = 0
被放到屏幕外
被其他层盖住
只在初始化闪一下后销毁
绑定到错误对象
```

现有 boot / usage 检查主要能证明“加载了、代码里有消费调用”，还不能证明“在正确场景、正确位置、正确状态下可见”。

### 3.3 generated 素材也容易被泛化绘制绕过

现在 generated / inline-svg 的消费证据主要看是否出现 `fillRect / strokeRect / arc / Graphics / <svg>` 等程序化绘制模式。这对“有程序化绘制”有效，但不能证明绘制的是某个 required slot。

例如一个色块游戏里，`ctx.fillRect()` 出现了，不等于每个 block 都按 `entity.color` 被正确绘制。

## 4. 建议新增一层：`visual-slots.yaml`

不要只让 assets.yaml 直接选素材。中间加一层视觉语义槽。

当前链路是：

```
asset-strategy
  -> assets.yaml
  -> implementation-contract
  -> registry
  -> business code
```

建议改成：

```
asset-strategy
  -> visual-slots.yaml
  -> assets.yaml
  -> implementation-contract
  -> registry/render wrapper
  -> runtime visual evidence
```

`visual-slots.yaml` 的职责是：**先定义游戏需要哪些视觉槽，再让 assets.yaml 去填槽。**

示例：

```
version: 1

slots:
  - id: board.block.fill
    required: true
    entity: block
    semantic-slot: color-block
    gameplay-fields: [color, durability]
    source-policy: generated-only
    visual-primitive: color-block
    allowed-asset-kinds: [generated-surface, svg]
    disallowed-asset-kinds: [tile, coin, gem, icon, button, sprite]
    render-zone: board-grid-cell
    state-driven:
      fill-color: entity.color
      label: entity.durability
    must-render: true
    allow-fallback: false

  - id: actor.pig.body
    required: true
    entity: pig
    semantic-slot: colored-character
    gameplay-fields: [color, ammo]
    source-policy: library-or-generated
    allowed-asset-kinds: [sprite, image]
    disallowed-asset-kinds: [button, tile, coin, panel]
    render-zone: conveyor-track
    state-driven:
      tint-color: entity.color
      badge: entity.ammo
    must-render: true
    allow-fallback: false

  - id: ui.waiting-slot.background
    required: true
    ui: waiting-slot
    semantic-slot: slot-surface
    source-policy: generated-or-panel
    allowed-asset-kinds: [panel, generated-surface, svg]
    disallowed-asset-kinds: [character, coin, tile]
    render-zone: bottom-hud
    must-render: true
    allow-fallback: false

  - id: fx.block.destroy
    required: false
    semantic-slot: destruction-fx
    source-policy: generated-or-particle
    allowed-asset-kinds: [particle, generated-surface]
    render-zone: board-grid-cell
    must-render: false
```

然后 `assets.yaml` 不再直接表达“我选了这个图”，而是表达“我用这个 asset 填了哪个 slot”：

```
images:
  - id: block-fill
    fulfills-slot: board.block.fill
    binding-to: block
    semantic-slot: color-block
    type: graphics-generated
    visual-primitive: color-block
    usage: 棋盘目标块颜色填充，颜色来自 block.color，耐久显示来自 block.durability

  - id: pig-body
    fulfills-slot: actor.pig.body
    binding-to: pig
    semantic-slot: colored-character
    type: local-file
    source: assets/library_2d/xxx/pig.png
    usage: 传送带上的小猪主体，运行时按 pig.color 着色并显示 ammo badge
```

这样 checker 可以判断：

```
这个 slot 需要 color-block
这个 asset 是否真的 color-block？
这个 slot 绑定 block.color
代码是否真的用 entity.color 驱动绘制？
这个 slot 需要出现在 board-grid-cell
runtime 是否真的记录它画在棋盘格？
```

这比 `binding-to: block` 强很多。

## 5. catalog 也要升级：从 pack-level 到 asset-level

你现在的 catalog 主要适合回答：

```
这个 pack 适合什么风格？
这个 pack 适合什么 genre？
```

但素材乱用需要回答的是：

```
这个具体文件能不能当 color-block？
能不能当 button？
能不能当 card surface？
能不能当 gameplay target？
能不能当 HUD icon？
```

建议给素材库增加 asset-level 或 family-level 元数据：

```
assets:
  - path: assets/library_2d/ui-pixel/button_001.png
    semantic-tags: [button, ui-control]
    asset-kind: button
    allowed-slots: [ui.button.background]
    disallowed-slots: [board.block.fill, actor.body, card.surface]

  - path: assets/library_2d/dungeon/tile_078.png
    semantic-tags: [terrain, dungeon-tile, floor]
    asset-kind: tile
    allowed-slots: [terrain.cell, scene.background-tile]
    disallowed-slots: [color-block, ui.button.background, gameplay-token]

  - path: assets/library_2d/items/coin_gold.png
    semantic-tags: [coin, collectible, currency]
    asset-kind: coin
    allowed-slots: [collectible.item, hud.currency-icon]
    disallowed-slots: [board.block.fill, ui.button.background, terrain.cell]
```

对于 Kenney 这类 tilemap 编号素材，建议每个 pack 旁边加一个轻量索引：

```
assets/library_2d/<pack>/_asset-index.yaml
```

例如：

```
tile_0013.png:
  asset-kind: terrain-tile
  semantic-tags: [grass, floor]
  allowed-slots: [terrain.cell, scene.background-tile]
  disallowed-slots: [color-block, ui.button, actor.body]

tile_0042.png:
  asset-kind: wall-tile
  semantic-tags: [wall, obstacle]
  allowed-slots: [terrain.obstacle]
  disallowed-slots: [color-block, button, hud-icon]
```

没有 asset-level 元数据时，checker 应该降级为“未知语义”，而不是默认通过：

```
unknown asset-kind + core slot = fail
unknown asset-kind + decor slot = warn
```

## 6. 改 contract：不要再对核心素材用 `inferRole`

当前 `generate_implementation_contract.js` 会从 id / usage / source 推断 role 和 kind；这适合作兼容旧流程，但不适合作核心 gate。

建议规则改成：

```
核心 slot：
  role / semantic-slot / asset-kind 必须来自 visual-slots.yaml + catalog，不允许靠 inferRole 推断。

非核心 / decor：
  可以 fallback 到 inferRole，但只能 warn，不能用于 must-render 通过。
```

新的 contract 应该长这样：

```
asset-bindings:
  - asset-id: block-fill
    slot-id: board.block.fill
    entity: block
    semantic-slot: color-block
    asset-kind: generated-surface
    type: graphics-generated
    source: generated
    binding-to: block
    gameplay-fields: [color, durability]
    render-zone: board-grid-cell
    render-as: procedural-surface
    must-render: true
    allow-fallback: false
    consumer: renderSlot
    evidence-required:
      - asset-bound
      - runtime-rendered
      - visible-on-screen
      - state-driven-color

  - asset-id: pig-body
    slot-id: actor.pig.body
    entity: pig
    semantic-slot: colored-character
    asset-kind: sprite
    type: local-file
    source: assets/library_2d/xxx/pig.png
    binding-to: pig
    render-zone: conveyor-track
    must-render: true
    allow-fallback: false
    consumer: renderSlot
    evidence-required:
      - asset-loaded
      - texture-created
      - runtime-rendered
      - visible-on-screen
```

这样 codegen 不是自由决定“怎么用素材”，而是必须满足 slot contract。

## 7. 改 usage 检查：从静态字符串升级到 per-asset 调用链

短期最值得改的是 `check_asset_usage.js`。现在它已经要求 required asset 100% 消费，这是对的；但要把“全局消费函数存在”改成“该 asset id 出现在消费调用里”。

可以先做正则版，不必一步到 AST：

```
function consumerPatternsForAsset(asset) {
  const id = escapeReg(asset.id);
  const base = asset.source ? escapeReg(basename(asset.source)) : null;

  if (asset.section === 'audio') {
    return [
      new RegExp(`getAudio\\(\\s*["'\`]${id}["'\`]`),
      new RegExp(`playSound\\(\\s*["'\`]${id}["'\`]`),
      new RegExp(`sound\\.add\\(\\s*["'\`]${id}["'\`]`),
      new RegExp(`sound\\.play\\(\\s*["'\`]${id}["'\`]`)
    ];
  }

  if (asset.type === 'graphics-generated' || asset.type === 'inline-svg') {
    return [
      new RegExp(`renderGeneratedPrimitive\\(\\s*["'\`]${id}["'\`]`),
      new RegExp(`renderSlot\\(\\s*{[^}]*assetId:\\s*["'\`]${id}["'\`]`, 's')
    ];
  }

  return [
    new RegExp(`getTexture\\(\\s*["'\`]${id}["'\`]`),
    new RegExp(`drawAsset\\(\\s*[^,]+,\\s*["'\`]${id}["'\`]`),
    new RegExp(`renderSlot\\(\\s*{[^}]*assetId:\\s*["'\`]${id}["'\`]`, 's'),
    new RegExp(`add\\.image\\([^)]*["'\`]${id}["'\`]`, 's'),
    new RegExp(`add\\.sprite\\([^)]*["'\`]${id}["'\`]`, 's'),
    new RegExp(`Sprite\\.from\\(\\s*["'\`]${id}["'\`]`)
  ];
}
```

然后把当前逻辑：

```
const hasRef = ...
const hasConsumer = consumerPatternForAsset(a).test(businessSrc);
if (hasRef && hasConsumer) used++;
```

改成：

```
const hasBoundConsumer = consumerPatternsForAsset(a).some((re) => re.test(businessSrc));
if (hasBoundConsumer) used++;
```

这样“id 字符串出现 + 另一个素材被 drawImage”不能再过。

## 8. 最好增加统一渲染 wrapper

真正稳的做法是禁止 required asset 直接裸用 `drawImage / add.image / new Sprite`，统一走一个 wrapper：

```
renderSlot({
  slotId: 'actor.pig.body',
  assetId: 'pig-body',
  entityId: pig.id,
  x,
  y,
  width,
  height,
  state: {
    color: pig.color,
    ammo: pig.ammo
  },
  draw(texture) {
    ctx.drawImage(texture, x, y, width, height);
  }
});
```

wrapper 内部记录：

```
window.__assetUsage.push({
  phase: 'render',
  slotId: 'actor.pig.body',
  assetId: 'pig-body',
  entityId: pig.id,
  semanticSlot: 'colored-character',
  renderZone: 'conveyor-track',
  x,
  y,
  width,
  height,
  visible: width > 0 && height > 0,
  stateDriven: {
    color: pig.color,
    ammo: pig.ammo
  },
  timestamp: performance.now()
});
```

Canvas / DOM / Phaser / Pixi / Three 都可以各自包一层：

```
canvas:
  registry.drawAsset(ctx, assetId, slotId, entityId, rect)

dom-ui:
  createAssetElement({ assetId, slotId, entityId })

phaser3:
  addAssetSprite(scene, { assetId, slotId, entityId, x, y })

pixijs:
  createAssetSprite(app, { assetId, slotId, entityId, x, y })

three:
  createAssetMesh(scene, { assetId, slotId, entityId, position })
```

然后新增：

```
check_asset_rendered.js
```

它不看源码字符串，而是用 Playwright 运行游戏后检查：

```
window.__assetUsage
```

必须满足：

```
每个 must-render asset:
  loaded = true
  bound to required slot = true
  rendered at least once = true
  visible = true
  width/height > 0
  slotId 与 contract 一致
  entityId 存在于 gameState
  renderZone 与 scene.yaml 对应
```

对于 DOM 还可以额外检查：

```
element.getBoundingClientRect()
getComputedStyle(element).display !== 'none'
getComputedStyle(element).visibility !== 'hidden'
Number(getComputedStyle(element).opacity) > 0
```

对于 Phaser / Pixi：

```
object.visible === true
object.alpha > 0
object.texture.key === assetId
object.getBounds() 在 viewport 内
```

这一步能直接解决“不用素材”问题。

## 9. 新增 `check_visual_semantics.js`

这个 checker 专门处理“乱用素材”。

输入：

```
specs/visual-slots.yaml
specs/assets.yaml
specs/implementation-contract.yaml
assets/library_2d/catalog.yaml
assets/library_2d/**/_asset-index.yaml
docs/game-prd.md
```

硬规则：

```
1. 每个 visual-core-entity 至少有一个 required visual slot。

2. 每个 required visual slot 必须有 asset fulfill。

3. asset.semantic-slot 必须等于 slot.semantic-slot 或在 allowed equivalent list 里。

4. asset-kind 必须在 slot.allowed-asset-kinds 内。

5. asset-kind 不得在 slot.disallowed-asset-kinds 内。

6. slot=颜色玩法实体时：
   必须声明 gameplay-fields 包含 color；
   必须 state-driven；
   local-file 只有 catalog 明确 allowed-slots 包含 color-block 才允许；
   否则必须 generated / inline-svg。

7. slot=button 时：
   asset-kind 必须是 button/panel/generated-surface；
   目标必须是 @ui；
   codegen 必须绑定真实 click handler。

8. slot=card-surface/text-bearing 时：
   禁止使用 button asset；
   禁止使用 coin/gem/icon 充当大面积文字承载面。

9. slot=actor.body 时：
   禁止 tile/button/panel/icon；
   必须 sprite/image/character family。

10. slot=background 时：
    不得满足 gameplay core entity 的 visual slot。
```

这比现在的 `binding-to` 检查更接近你真正想要的“素材用在合适的位置”。

## 10. 把几个 warning 升级成 fail

我建议先改这几条，收益很高：

### 10.1 `decor` 占比在核心游戏里应该 fail

当前 `decor > 40%` 是 warning。建议改成：

```
如果 visual-core-entities 非空：
  任一 core entity 的 asset 不允许 binding-to: decor
  local-file decor ratio > 40% -> fail
否则：
  decor ratio > 40% -> warn
```

### 10.2 strict 风格下 pack coherence 应该 fail

当前 pack coherence 低于阈值也是 warning。建议：

```
style-coherence.level = strict:
  top pack ratio < threshold -> fail

style-coherence.level = flexible:
  top pack ratio < threshold 且无 pack-mix-reason -> warn

style-coherence.level = n/a:
  skip
```

### 10.3 required slot 没有 must-render 不能只是 warn

现在如果 contract 里没有 `must-render=true`，`check_asset_usage.js` 会 warning 后结束，理由是“也许所有 asset 都是 decor”。建议改成：

```
如果 PRD asset-strategy.visual-core-entities 非空，
并且 visual-slots 里有 required slot，
但 contract 中 must-render=true 的 asset 为空：
  fail
```

这能防止“素材选了但都被降级成非核心 / decor”。

## 11. 对 Pixel Flow 这类游戏的具体素材策略

Pixel Flow 这类玩法里，不应追求“所有东西都用素材库”。正确策略应该是混合：

```
棋盘色块：
  generated color-block
  因为 color 是玩法字段，必须精确表达。

小猪主体：
  local-file sprite 或 generated character
  可以用素材库，但需要 runtime tint / badge 表示颜色和数值。

等待槽 / 传送带 / HUD：
  generated panel / local-file UI panel
  重点是清晰，不要用 dungeon tile 或 coin 代替。

按钮：
  button/panel/generated-surface
  必须绑定真实 click handler。

特效：
  generated particle / simple tween
  不应拿 coin/gem 当爆炸碎片，除非 catalog 明确标成 particle。
```

也就是说，**不是 local-file 越多越好**。对抽象玩法对象，generated 反而更正确。你现在已有 `generated-only` 和 `visual-primitive: color-block` 的思路，应该把它扩大成 slot-level 策略，而不是只对色块做特判。

## 12. 最小落地顺序

### P0：一两天内能改，直接堵“不用素材”

1. 改 `check_asset_usage.js`：required asset 必须出现在与自己 id 绑定的消费调用里，不能再用“id 字符串 + 全局 drawImage/add.image”通过。
2. 增加 `renderSlot()` wrapper，先支持 canvas / dom-ui，Phaser/Pixi 后续补。
3. 新增 `window.__assetUsage`，记录 `assetId / slotId / entityId / x / y / w / h / visible`。
4. 新增 `check_asset_rendered.js`，先只检查 `must-render=true` 的 asset 是否有 runtime render 记录。

### P1：三到五天内改，解决大部分“乱用素材”

1. 新增 `visual-slots.yaml`。
2. assets.yaml 增加：

```
fulfills-slot:
semantic-slot:
asset-kind:
state-driven:
```

1. `generate_implementation_contract.js` 对 required asset 不再靠 `inferRole`，而是从 slot 和 catalog 取语义。
2. `check_asset_selection.js` 增加 slot 匹配硬规则。
3. `decor > 40%` 在核心实体非空时从 warning 改 fail。

### P2：一周级别，提升素材选择质量

1. 给素材库加 `_asset-index.yaml`。
2. 对 tilemap 编号素材做 asset-level 标签。
3. `selection-report` 改成 decision table：

```
selection-report:
  slot-decisions:
    - slot: actor.pig.body
      selected: pig-body
      candidates:
        - asset: pig-body
          score: 0.91
          reason: sprite + same pack + style match
        - asset: coin-gold
          rejected: true
          reason: asset-kind coin not allowed for actor.body
```

1. 增加 `check_visual_semantics.js`，作为 Phase 3 gate。

## 13. 结论

素材问题的核心不是“模型不认真选”，而是当前链路仍然允许它把素材当成普通文件处理。

要解决乱用：

```
binding-to -> semantic-slot / visual-slot
pack-level catalog -> asset-level allowed-slots
inferRole -> explicit slot contract
warning -> core slot fail
```

要解决不用：

```
id 出现 + 消费函数出现
  -> assetId 绑定到具体消费调用
  -> renderSlot runtime evidence
  -> visible-on-screen evidence
```

这是一轮**局部重构**，不需要推翻小游戏链路。先改 `check_asset_usage.js` 和加 `renderSlot/__assetUsage`，会立刻改善“不用素材”；再加 `visual-slots.yaml` 和 asset-level catalog，才能系统性解决“素材乱用”。

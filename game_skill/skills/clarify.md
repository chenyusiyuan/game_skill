# Phase 1.x: Clarify（按需触发）

## 职责

brief 扫描发现以下任一情况时触发：

- **≥ 2 个 GamePRD 必填字段缺口**
- **功能优先级不清**
- **用户预期交付档位不清**
- **强约束与 runtime 能力矛盾**
- **3D / 2D 维度模糊**（query 出现"立体/空间"但未明确 3D/Three.js）

目标是用**最少的提问**补齐产品边界，而不是穷举用户意图。具体功能机制（触发时机、目标选择、作用范围、资源消耗时点、移动/碰撞粒度等）不在 Phase 1 追问，统一等 GamePRD 写完后进入 `spec-clarify.md`。

---

## 触发条件（由 prd.md 的 Step 2 判定）

以下任一满足 → 运行本 skill；否则跳过。

- Brief 的 Gaps 列表 ≥ 2 项
- 用户明确写了 3 个及以上功能/系统，但没有说明哪些是必须保留
- 用户描述是「像 X 一样」/「做个 roguelike」/「做个经营游戏」这类大类需求，但核心功能版本差异很大
- 用户给了长规格，但没有说明这次想要 `prototype / playable-mvp / feature-complete-lite` 哪种交付目标
- hard-rule 与候选 runtime 矛盾（如用户要「实时物理」但提到要 DOM-only）
- 参考游戏歧义（如「俄罗斯方块」可能指经典版或主题变体）

---

## 提问原则

### 数量

- **最多 3 个问题**，一次性发出
- 每个问题提供 2-5 个选项（AskUserQuestion 模式），含一个明显推荐选项
- **每个问题必须提供"让我决定"选项**，通常放最后，文案为"让我决定（按当前 query 自动选择）"或更具体的默认策略
- 用户选择"让我决定" / "你决定" / "默认" / "随便"时，不再追问；按当前 Inferred + genre 默认表给出保守决策，并把该决策写入 ClarifiedBrief
- 用户选错也不是大事——后面可以在 Phase 2 PRD 中调整

### 优先级

按「影响下游决策的不可逆程度」排序。高优先级问题：

1. **genre 不明确**：影响 runtime 选择、章节模板、校验 profile
2. **核心玩法循环不明确**：影响 `@flow` + `@scene` 结构
3. **胜负条件不明确**：影响 `@rule` 和 `@check`
4. **必须保留的功能不明确**：影响 §3 范围、Phase 3 specs、Phase 4 codegen
5. **交付档位不明确**：影响 strategy 是否默认走最小闭环，还是尽量贴近用户预期
6. **关卡/内容规模不明确**（如游戏内容量）：影响 `@resource` / `@level`

低优先级（通常 Phase 2 可以自己决定，不问）：
- 纯表现性细节（字体、次要配色、装饰动画）— 由 color-scheme 自动推断
- 具体分数公式（给个默认值）
- 具体功能机制细节（如攻击/碰撞触发粒度、索敌范围、回收时机、技能结算顺序）— 后置到 `spec-clarify.md`
- 技术细节（runtime 具体版本、包管理器）
- 视觉风格 — **已移至 `semantic-clarify.md`**，仅在结构清晰、仍有提问额度且风格会影响素材选择时，按需推荐 Top-3 色板供用户选择

## 问题模板

### 模板 1：genre 不明确

> "你想做的更像以下哪一类小游戏？"
> - A. 棋盘/格子规则型（贪吃蛇、消消乐、俄罗斯方块）
> - B. 控制闯关型（跳跃、物理解谜）
> - C. 经营养成（电子宠物、clicker）
> - D. 其他（填写）
> - E. 让我决定（按 query 语义选择最稳的 genre）

### 模板 2：玩法循环不明确

> "游戏的一局从开始到结算要经历哪几步？"
> - A. 我来描述（自由文本）
> - B. 让我决定（按 genre 生成一个标准核心循环）

### 模板 3：胜负条件不明确

> "怎样算通关？"
> - A. 在限时内完成目标
> - B. 收集/消除到指定数量
> - C. 存活到最后
> - D. 其他（填写）
> - E. 让我决定（按玩法类型选择最常见胜负条件）

### 模板 4：功能优先级不明确

> "这一版哪些功能必须保留？最多选 3 项"
> - A. 核心战斗/核心配对/核心操作循环
> - B. 成长或养成系统（职业/装备/升级/经济）
> - C. 关卡与内容量（多关卡、更多词库、更多敌人）
> - D. 强反馈表现（动画、粒子、手感）
> - E. 让我决定（优先保核心闭环，再保留最相关的 1-2 个系统）

### 模板 5：交付档位不明确

> "这次你更想要哪种结果？"
> - A. `prototype`：先验证玩法概念，允许很多功能先不做
> - B. `playable-mvp`：最小可玩闭环，优先跑通主流程
> - C. `feature-complete-lite`：尽量保留你点名的核心功能，只压缩内容量和深度
> - D. 让我决定（短 query 默认 playable-mvp，长规格默认 feature-complete-lite）

### 模板 6：裁剪策略不明确

> "如果实现复杂度超预算，你更希望我优先怎么裁剪？"
> - A. 先缩内容量，不先砍核心系统（推荐）
> - B. 先保留画面和反馈，系统可以简化
> - C. 先保留系统完整性，视觉和内容量可以弱化
> - D. 让我决定（优先保留可玩闭环和用户点名的硬要求）

### 模板 7：强约束与能力矛盾

> 这里有个冲突：你要求 `<hard-rule>`，但 `<runtime>` 不擅长这个场景。你倾向于：
> - A. 换引擎（推荐 `<alt-runtime>`，完美支持该需求）
> - B. 保留 `<runtime>` 但放宽这条约束
> - C. 让我自己权衡（我会在 Phase 2 给降级方案）
> - D. 让我决定（优先保证硬规则可落地，必要时换到推荐引擎）

### 模板 8：3D / 2D 维度不清

仅当用户 query 同时具备"立体/空间/模型"等模糊信号，但又没明确说"3D / Three.js / 第一人称"时触发。

> "这个游戏你希望是 2D 还是 3D？"
> - A. 2D（推荐）：用 Phaser/PixiJS/Canvas/DOM，LLM 生成成功率最高，素材库已就绪
> - B. 3D：用 Three.js，适合第一人称、走迷宫、3D 平台跳跃、3D 模型展示；当前 3D 素材库建设中，可能回退到几何体占位
> - C. 让我决定（按 query 语义判断；默认走 2D）

**回填规则**：
- 选 A 或 C → `is-3d: false`，引擎候选仅限 2D 四件套
- 选 B → `is-3d: true`，引擎候选仅 `three`

---

## 输出

将用户回答直接回填到 `docs/brief.md` 的 `ClarifiedBrief` 段，包括但不限于：

- `must-have-features`
- `nice-to-have-features`
- `delivery-target`
- `cut-preference`

然后：

- 若所有 gap 都补齐 → `.game/state.json` 的 `understand.status = "completed"`
- 若用户回答后又冒出新 gap → 最多再问 1 轮（累计不超过 3 个 round，防止无限澄清）
- 若 3 轮后仍不清晰 → `understand.status = "failed"`，task_done 要求用户给更具体的需求

---

## 不要做的事

- ❌ 在核心功能优先级没搞清楚前，先去问配色或视觉风格（视觉风格由 `semantic-clarify.md` 处理）
- ❌ 默认把用户点名的系统当作「nice-to-have」
- ❌ 在交付档位不清时直接按最小 MVP 处理
- ❌ 问 runtime 具体版本——engine guide 会处理
- ❌ 在 Phase 1 追问 trigger / condition / effect / target-selection / timing 这类机制细节——后置到 `spec-clarify.md`
- ❌ 问对方已经说清楚的东西
- ❌ 一次提超过 3 个问题
- ❌ 连环追问细节（用户会烦）
